\subsection{Detailed Description}
Interface definitions for configuration file loader. \begin{DoxyAuthor}{Author}
Lucas Kramer
\end{DoxyAuthor}
The format of the config file is a series of lines with the format\par
 $<$type$>$ $<$name$>$ = $<$value$>$\par
 Blank lines are allowed, and comments \# can occur at the end of a line. \par
 Valid types are int, hex, octal, float, bool/boolean, char, and string. Note that int, hex, and octal all create a variable with type int, and only affect how the value is parsed. \par
 Names can be \mbox{[}A-\/\-Za-\/z\mbox{]}\mbox{[}A-\/\-Za-\/z0-\/9\-\_\--\/\mbox{]}$\ast$, but the recommended usage is all upper case, seperated by underscores. \par
 The following is a list of valid value formats\-: \begin{TabularC}{4}
\hline
{\bfseries Type name} &{\bfseries Format} &{\bfseries Examples} &{\bfseries Notes}  \\\cline{1-4}
int &\mbox{[}0-\/9\mbox{]}+ &42  \\\cline{1-4}
hex &(0x)?\mbox{[}0-\/9\-A-\/\-Fa-\/f\mbox{]}+ &0x3\-A\-F4, 0x3af4, 3\-A\-F4 &It is recommended to use the 0x prefix and upper case A-\/\-F  \\\cline{1-4}
hex &\mbox{[}0-\/7\mbox{]}+ &0123, 123 &It is recommended to use the 0 prefix  \\\cline{1-4}
float &\mbox{[}0-\/9\mbox{]}+(.\mbox{[}0-\/9\mbox{]}+)? &3, 3.\-14  \\\cline{1-4}
bool/boolean &true$\vert$false &true, false &It is recommended to use the bool type name  \\\cline{1-4}
string &"\mbox{[}$^\wedge$"\textbackslash{}n\mbox{]}$\ast$" &\char`\"{}\-Hello, World!  \char`\"{}, \char`\"{}\char`\"{}, \char`\"{}\$\-O\-T\-H\-E\-R\char`\"{} &\$ followed by a variable is expanded to that variable. An actual \$ must be escaped  \\\cline{1-4}
char &'(\mbox{[}$^\wedge$\textbackslash{}n\textbackslash{}r\textbackslash{}t\mbox{]}$\vert$\textbackslash{}\textbackslash{}n$\vert$\textbackslash{}\textbackslash{}r$\vert$\textbackslash{}\textbackslash{}t)' &'a', 'A', '4', '\textbackslash{}n', '\textbackslash{}t'  \\\cline{1-4}
\end{TabularC}
\par
 In addition to variable settings, it is possible to load other configuration files. This has the syntax\-:\par
 use $<$filename$>$\par
 where filename is a string. The path to $<$filename$>$ is with respect to the directory containing the current file -\/ absolute paths are not permitted. When an included file contains a variable with the same name as a value that is already defined, the current behavior is to overwrite the old value. Any value re-\/definition causes a warning.
